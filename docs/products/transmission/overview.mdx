# Transmission

Transmission is ACME's dynamic model router. It classifies each task and routes it to the model best suited to handle it — automatically, with sub-10ms overhead, and full observability.

Most agent systems use static model configuration: one model for everything, or a hand-coded fallback chain. Transmission replaces that with task-aware routing. A quick lookup routes to a fast, cost-effective model. A complex reasoning task routes to a more capable one. The routing adapts to the work — not to whoever last edited the config file.

## The Problem With Static Routing

One model for everything sounds fine until you notice your expensive model is handling simple lookups, and your cheap fallback is getting routed complex reasoning tasks because that's just how the chain shakes out.

Transmission replaces static fallback chains with routing that knows what the task actually is. Simple task, efficient model. Complex reasoning, capable model. You stop paying for mismatches and stop wondering why something went wrong.

## How Routing Works

Transmission resolves routing through four layers in order:

### 1. Work Class Resolution

Transmission determines task type using the highest-confidence source available:

| Priority | Source | Example |
|----------|--------|---------|
| 1 | Dispatch hint (if provided) | Workflow system declares `work_class: coding` |
| 2 | Explicit work class | Caller declares task type directly |
| 3 | Agent metadata | Session context includes task classification |
| 4 | Heuristic classifier | Keyword analysis of the prompt |
| 5 | Lane default | Falls back to lane-appropriate baseline |

### 2. Lane

Tasks run in one of two lanes:

| Lane | Optimizes For | Default Preference |
|------|--------------|-------------------|
| `interactive` | Quality and capability | Premium → budget-capable → mid → efficient |
| `background` | Cost and throughput | Budget-capable → mid → premium → efficient |

### 3. Confidence Gearing

If the classifier confidence falls below the configured threshold (default: 70%), Transmission gears up — routing to a higher-capability model rather than committing to a low-confidence classification.

This prevents a misclassified complex task from landing on an underpowered model.

### 4. Execution Config

Transmission also resolves execution parameters — streaming mode, temperature, tool-calling, structured output — based on work type, and masks any settings the selected model doesn't support.

## Routing Log Format

Every decision is emitted as a structured NDJSON event:

```json
{
  "ts": "2026-03-15T19:30:00.000Z",
  "req_id": "tr-a3f2b1c4",
  "event": "TRANSMISSION_SUCCESS",
  "model": "openrouter/deepseek/deepseek-v3.1-terminus:exacto",
  "work_class": "coding",
  "confidence": 0.83,
  "classifier_source": "heuristic",
  "gear": "budget-capable",
  "duration_ms": 1.42
}
```

Log path: `~/.openclaw/watchdog/transmission_events.log`

## Performance

Transmission is designed to add no meaningful overhead to the request path:

| Metric | Value |
|--------|-------|
| p95 routing latency (heuristic path) | < 10ms |
| Classification cache | 100-entry LRU, keyed on prompt prefix |
| Blocking I/O in hot path | None |
| Writes to agent config files | Never |

## What Transmission Does NOT Do

- Does not make model API calls — it routes to the model, it doesn't invoke it
- Does not modify your `openclaw.json` or any agent configuration
- Does not handle failover or continuity — that's your model fallback chain
- Does not replace your model chain config — it decides which entry in the chain to use
