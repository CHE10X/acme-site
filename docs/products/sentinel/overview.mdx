# Sentinel

Your agent can be running and broken at the same time. Sentinel watches for that.

It runs continuously alongside your stack and tells you when something is wrong — before you find out the hard way.

<Warning>
Sentinel is **read-only and advisory**. It watches and alerts. It never restarts anything, changes your configuration, or takes action on your behalf.
</Warning>

## What Sentinel Watches For

**Silence gaps** — your agent stopped responding when it shouldn't have. Not a crash — just... nothing happening anymore.

**Stall patterns** — the process is alive, the port is up, but work stopped. Sentinel knows the difference between "running" and "actually doing something."

**Memory pressure** — compaction acceleration that historically shows up before a multi-minute freeze. You get a warning while you can still act.

**Log growth** — disk pressure building quietly in the background, caught before it becomes a storage problem.

## What You'll See

Sentinel maintains four states and tells you which one you're in:

| State | What It Means |
|-------|--------------|
| `NOMINAL` | Everything looks normal |
| `SUSPECT` | Early signals — worth keeping an eye on |
| `ACTIVE` | Something's wrong — check your stack |
| `STORM` | Multiple problems overlapping — act now |

You don't have to poll for this. When the state changes, Sentinel logs it. Agent911 reads it. You find out.

## How It Fits In

Sentinel is built into the Watchdog loop — every heartbeat check carries Sentinel's current read of your stack. That means:

- When Watchdog probes your gateway, it already knows if Sentinel is seeing pressure
- A `STORM` state shows up everywhere: `watchdog.log`, `ops_events.log`, Agent911
- You get one consistent picture, not three different dashboards to reconcile

## Files It Writes

Sentinel writes three files, all read-only outputs — it never touches your config:

| File | What's In It |
|------|-------------|
| `compaction_alert_state.json` | Current memory pressure level and rate |
| `sentinel_protection_state.json` | Current state, when it changed, what triggered it |
| `ops_events.log` | Running log of every state change |

## When Something Goes Wrong

If Sentinel hits `ACTIVE` or `STORM`:

1. Run `triage` first — capture a snapshot before touching anything
2. Open Agent911 — it'll show you which signals are active
3. Check `sentinel_protection_state.json` if you want the raw detail

Don't restart first and diagnose second. That's how you lose the evidence.
